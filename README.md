# huffman-handson

Go でハフマン符号（Huffman coding）を一から実装して、可逆圧縮の仕組みを体験するハンズオン用のリポジトリです。
実際のファイルを読み込んで圧縮・復元し、圧縮率を確認できます。

## これは何？

文章の中で **よく出る文字は短いビットで、めったに出ない文字は長いビットで** 表すことで、
データ全体のビット数を減らす **ハフマン符号** を Go で実装しています。

`main.go` 一本に、圧縮（符号化）と展開（復号）の全工程が入っています。

## 使い方

任意のテキストファイルを渡すと、`<ファイル名>.huff` に圧縮して書き出し、
その圧縮ファイル単体から復元して元と一致するか検証します。

```bash
go run main.go sample.txt
```

出力例:

```
入力ファイル  : sample.txt
圧縮ファイル  : sample.txt.huff
復号は正しい? : true
--- 圧縮効果（実バイト・木込み）---
元ファイル    : 416 バイト
圧縮ファイル  : 401 バイト（木の情報を含む）
削減率        : 3.6%
```

## 仕組み（4ステップ）

| 手順 | やること | 対応する処理 |
|------|----------|--------------|
| 1 | 各文字の**出現頻度**を数える | `compress` 内の `freq` |
| 2 | 頻度の小さい2つを合体して**ハフマン木**を作る | `buildTree` |
| 3 | 木を再帰的にたどって**符号表**を作る | `buildCodes` |
| 4 | 符号表で**符号化**し、木をたどって**復号**する | `encode` / `decode` |

ビット列は `packBits` / `unpackBits` で本物のバイトに詰め直し、
圧縮ファイルには復元用の**頻度表も一緒に保存**（`encoding/gob`）するため、
`.huff` ファイル単体で元に戻せます。

## ポイント1：なぜ迷わず復号できるのか

ハフマン符号は **接頭符号（prefix code）** — どの文字の符号も別の文字の符号の先頭に
ならない — なので、区切り記号なしのビット列でも木を根からたどるだけで一意に復元できます。
文字は必ず木の「葉」にしか置かれないため、この性質が構造的に保証されます。

## ポイント2：小さいファイルは縮まないことがある

復号には木（頻度表）が必要で、圧縮ファイルにはその情報も含めます。
そのため**小さいファイルでは木の保存コストが圧縮の利得を食いつぶし**、
ほとんど縮まない（場合によっては増える）ことがあります。
ファイルが大きいほど木のコストは相対的に無視でき、圧縮率が上がります。

```
sample.txt (416B)   → 3.6% 削減（木のコストが重い）
sample.txt ×50 (20KB) → 46% 削減（木のコストが薄まる）
```
